# SSANM Audit / Fix Ledger

## Current hypothesis

The first two current JSON-AS SSANM-only runtime failures are stale-alias bugs in the raw/LocalGraph rewrite boundary for AssemblyScript root-stack spill shapes. Earlier repairs only corrected aliases when a spilled argument read was redirected to a future scratch local. The latest reduced functions showed (1) the opposite sequential-spill hazard, where a read immediately after a stack spill still pointed at the previous spilled local instead of the just-spilled local, and (2) a stable call-result local read retargeted to a freshened scratch alias. Both pass validation and broad fuzzing because the wasm is well-typed but reads the wrong AssemblyScript GC/root-stack value at runtime.

After fixing those two families, the same JSON-AS artifact still failed at runtime in `OBJECT#get:rtSize` / string escaping. Function-swap bisection first pointed at a larger parser function (`defined=980`, printed `#func984`) with loop/result/default-materialization drift. A focused reduction confirmed one structural sub-family: LocalGraph get decisions could be consumed by raw structured/default application at a different syntactic local than the plan entry, so a branch-value block write could be freshened while its immediate reads stayed canonical or were materialized as zero. The patch now guards get-decision application by `original_local`, keeps writes inside plain-`br` value blocks canonical, and tracks already-written locals before default materialization.

After that parser-load fix, direct runtime still failed. Prefix function-swap bisection against Binaryen then identified `defined=1185`, printed `#func1189`, `~lib/util/number/itoa32`: raw structured rewriting freshened branch-local allocation `local.tee $out` writes, but the suffix still read canonical `$out`, producing a null/stale string pointer. The current patch treats a dropped base-local write with a later surviving canonical read as unsafe when the original shape has both arms of a void `if` writing that local and a suffix read before the next write, forcing the safe fallback for this no-merge merge-read shape. The JSON-AS whitespace SIMD artifact now validates and passes the Node WASI runtime replay after direct `--ssa-nomerge`.

## Confirmed invariants

- SSANM must preserve observable behavior immediately after the pass; later cleanup must not be required for validity or semantic repair.
- For `local.set x <rhs>`, the RHS reads the pre-set version of `x`; the write becomes current only after RHS evaluation.
- For `local.tee x <rhs>`, the RHS is evaluated once, the result is written to `x`, and the same value remains available to the surrounding expression.
- Binaryen `SSAify(false)` is a no-merge pass: freshen only explicit writes whose influenced gets stay single-source; preserve canonical locals for multi-source merge reads; do not create full-SSA merge locals for no-merge mode.
- Entry parameter/default values are real reaching sources. One-arm branches that merge an explicit write with entry/default must stay canonical.
- Branch values, loop backedges, `br_if` splits, `br_table`, EH, typed-control, and unreachable/polymorphic states are merge/control boundaries unless explicitly proven safe.
- Raw stack-spill repairs must not clone, drop, or move calls/stores/trapping expressions; they may only retarget local indices in already-linear local traffic when the alias relation is structurally proven.

## Binaryen `SSAify(false)` comparison notes

- Local source refreshes already in this repo (`docs/wiki/raw/binaryen/2026-06-13-ssa-nomerge-version-130-source-refresh.md`) record that local `wasm-opt` is `version_130` and that `src/passes/SSAify.cpp` did not drift from the earlier `version_129` no-merge contract.
- Binaryen's no-merge mechanism is LocalGraph-driven: `SSAify.cpp` freshens a write only when `!graph.isSSA(set->index) && (allowMerges || !hasMerges(set, graph))`; for `ssa-nomerge`, `allowMerges` is false, so merge-feeding writes stay on canonical locals.
- Binaryen `SSAify(false)` does not materialize full-SSA merge locals for multi-source gets. Full `ssa` (`SSAify(true)`) owns the merge-local / `local.tee` / entry-prepend machinery; no-merge leaves those reads canonical.
- The two fixed JSON-AS failures are not Binaryen merge-local behavior. They are Starshine-local raw/planned rewrite hazards around the chosen fresh-local representation for AssemblyScript root-stack spills. Binaryen's output for the same functions keeps the correct operand locals after freshening: `func858` uses the just-spilled local for the next live operand, and `func851` keeps the stable call-result local for `len`.
- Relevant Starshine contrast: `src/passes/ssa_nomerge.mbt` builds the no-mutation LocalGraph plan, while `src/passes/pass_manager.mbt` applies raw fast-path rewrites for many WAT/AssemblyScript-shaped functions. The current fixes harden that raw apply phase; they do not change Binaryen's no-merge policy.

## Remaining risk

- The latest direct JSON-AS whitespace SIMD SSANM-only artifact passes: `target/native/release/build/cmd/cmd.exe --ssa-nomerge --out .tmp/ssanm-audit-current-fixed7-whitespace-simd.wasm .tmp/json-as-f707d68/.as-test/build/simd/whitespace.spec.wasm`, `wasm-tools validate --features all`, and `node .tmp-run-wasi.mjs ../../.tmp/ssanm-audit-current-fixed7-whitespace-simd.wasm` from `.tmp/json-as-f707d68` all completed successfully. The source-level cloned json-as smoke also passes: `npx ast run assembly/__tests__/whitespace.spec.ts --config as-test.node.config.json --mode simd --enable try-as --clean` reported `62` tests passing in the SIMD mode.
- The branch-allocation fix is deliberately conservative: it rejects unsafe raw structured outputs instead of teaching the fast path to preserve every `itoa32` merge write. The current gate is structural (void `if` branches both write a local, then a suffix reads it before the next write); a future refinement could use explicit per-local reaching-write proof to allow more fast-path cases.
- The raw SSANM fallback still contains several artifact-derived local-pattern repairs. Each should be treated as a narrow proof obligation, not a general dataflow solution.
- Full post-pass dominance/reaching-definition validation is not yet implemented in this audit slice.
- Plain `br`, `br_table`, branch values, EH, typed loops/results, and broad nested branch targets remain fail-closed or helper-owned surfaces; they need separate positive expansion before mutation.

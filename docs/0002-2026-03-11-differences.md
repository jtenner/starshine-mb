# Optimize Pipeline Differences vs. Binaryen

Status: complete comparison pass with key parity points and remaining runtime gaps.

## Scope
- Compare:
  - Binaryen `PassRunner` defaults
  - local `src/passes/optimize.mbt`
- Sources:
  - local function/global pipelines
  - key scheduler helpers and segment runners

## High-Level Findings
- Pass ordering is intentionally aligned at the high level.
- Differences are mostly in:
  - execution model,
  - validation timing policy,
  - and specific pass semantics (`Vacuum`, `StringGathering`, partial `cfp-reftest` parity).
- A strict pass-list parity check is not sufficient to explain observed pathological runtime.

## Pass-Sequence Parity Snapshot
- `ssa-nomerge` parity: **closed** (`SSANoMerge` now explicit, no longer routed through `DataflowOptimization`).
- `RemoveUnusedNames` cleanup spacing: **closed**.
- second `OptimizeInstructions` decision: **intentional divergence** (runtime guard skips late pass on large modules unless shrink asks for it).
- `Precompute`/`PrecomputePropagate` gating: **closed**.
- `CodePushing`/`CodeFolding` gating: **closed**.
- `CFP` family: **split** into
  - `ConstantFieldPropagation` + `ConstantFieldNullTestFolding`
  - no longer claims full `cfp-reftest` parity.
- `InliningOptimizing` + `Directize` in global post: **present** with intended mode.
- `StringGathering`: **not yet in pipeline** (known feature omission).

## Global Pre-pass Status
- `DuplicateFunctionElimination` + `RemoveUnusedModuleElements` + `MemoryPacking` + `OnceReduction` now follow expected structure.
- GC/closed-world sequence tracks Binaryen intent:
  - `TypeRefining` / `SignaturePruning` / `SignatureRefining`
  - `GlobalRefining` / `GlobalTypeOptimization`
  - `ConstantFieldPropagation` + `ConstantFieldNullTestFolding`
  - `GlobalStructInference` / `AbstractTypeRefining` / `Unsubtyping`
- Remaining differences are mainly narrow or missing feature pass implementations.

## Runner and Validation Differences
- Local runner now uses function-pass stacks + segment boundaries.
- It still differs from Binaryen by lacking worker-parallel function scheduling and by different validation policy defaults.
- default `OptimizeValidationPolicy` is final-only validation (less overhead than per-pass validation).
- `--after-segment` validation exists as a stricter debug mode.
- `Vacuum` includes a repeated no-op skip guard not present in Binaryen default ordering.

## What This Means for Pathology Analysis
- Parity mismatch is **not** just pass-list mismatch.
- Main investigation areas for runtime are:
  1. function scheduling model,
  2. heavy-module and cache behavior in pass implementations,
  3. and strictness of validation placement.
- `StringGathering` and `ConstantFieldNullTestFolding` are completeness gaps; `StringGathering` is the only explicit missing global post pass in the default path.

## Suggested Use
- Interpret parity in four layers:
  1. pass sequence,
  2. IR-shaping behavior,
  3. runner/runtime model,
  4. pass-internal complexity.
- This document is the baseline for future performance and completeness tracking.

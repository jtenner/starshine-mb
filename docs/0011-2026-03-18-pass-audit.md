# Pass Audit

Status: baseline pass inventory and performance ordering checkpoint.

## Purpose
Track pass sequencing, implementation parity, and optimization risk ranking against Binaryen-inspired behavior.

## Scope
- `src/passes` implementations and runner pipeline.
- `src/passes/optimize.mbt` ordering and validation policy.
- Related docs and benchmark evidence.

## Current Picture
- Pass list and scheduler coverage are now explicit in:
  - function pre-pass,
  - function stack pipeline,
  - global pre-pass,
  - global post-pass.
- Runner is segmented into `FunctionPassStack` and `BarrierPass` blocks.
- Module-wide generated pass execution now threads `PipelineFeatures` at runtime.
- Generated optimize pipeline expansion and execution now derive `has_gc` and `has_multivalue` from the input module and thread `closed_world`, `zero_filled_memory`, `low_memory_unused`, and `traps_never_happen` from `OptimizeOptions`.
- `MemoryPacking` now has a dedicated generated-runner dispatch surface, the documented analysis-only entry gating for unsupported memory topologies and unsafe active/imported-memory layouts, the initial segment-op pre-normalization for obvious `memory.init` / active `data.drop` cases, per-`DataIdx` referrer collection for both bulk-memory and GC data consumers, early passive-segment compaction with `data.drop -> nop`, `DataIdx` / `data_names` remap, and `DataCntSec` updates, range analysis for split eligibility, thresholded zero runs, startup-trap preservation, and the segment-count guardrail, full segment materialization with checked active-offset shifting, the typed passive replacement walk for `memory.init` / `data.drop` plus dropped-segment state tracking and temp-local allocation, and explicit validation/idempotence coverage on the rewritten outputs. `MemoryPacking` is no longer an active default-pipeline blocker.
- Default validation is not every-pass; final-module validation is available by policy.

## What Is Mostly Aligned
- Major pipeline shape matches Binaryen no-DWARF defaults.
- `RemoveUnusedModuleElements` is now implemented across its intended open-world and closed-world direct-consumer scope:
  - exported/start roots, live-function traversal, and function-index compaction are implemented,
  - defined-global liveness and remap are now implemented, including mixed import+defined `GlobalIdx` rewriting and `global_names` compaction,
  - defined-table and defined-memory liveness/remap are now implemented, including mixed import+defined `TableIdx`/`MemIdx` rewriting and `table_names`/`memory_names` compaction,
  - defined-tag liveness/remap is now implemented, including mixed import+defined `TagIdx` rewriting and `tag_names` compaction,
  - `elem`/`data` segment liveness compacts passive segments, rewrites `ElemIdx`/`DataIdx` users plus `DataCntSec`, keeps active segments only when their targets are imported or otherwise live, roots active segments whose offsets are unknown or definitely out-of-bounds for the defined target’s initial bounds only when traps may happen, lets indirect-call tables retain only matching active `elem` contributors, no longer lets `table.size` or pure table writes pin prior active elem contents, and only scans table initializer expressions when table contents are actually observable or participate in indirect calls,
  - closed-world `ref.func` / `call_ref` handling now distinguishes referenced functions from used functions and rewrites referenced-only defined bodies to `unreachable`,
  - mutated indirect tables now feed their callable signatures into the same closed-world function-reference liveness machinery,
  - descriptor-bearing constant-initializer trap rooting is now implemented for nullable `struct.new_desc` / `struct.new_default_desc` initializers,
  - and closed-world typed `struct.new` / `struct.new_desc` field payloads plus `struct.new_desc` / `struct.new_default_desc` descriptor operands, mutable `struct.set`, ref-bearing `array.new` / `array.new_fixed` / `array.new_elem` payloads, mutable `array.set` / `array.init_elem` writes, ref-bearing `array.fill`, ref-bearing `array.copy`, reference-identity consumers (`ref.is_null`, `ref.as_non_null`, `ref.test`, `ref.cast`, `ref.eq`, `ref.get_desc`, `ref.test_desc`, `ref.cast_desc_eq`, `br_on_null`, `br_on_non_null`, `br_on_cast`, `br_on_cast_fail`), conversion-style wrappers (`extern.convert_any`, nested `any.convert_extern`), and direct no-read carrier globals (`drop(global.get(...))`, `array.len(global.get(...))`) now stay reference-only until a live `struct.get*` or `array.get*` read makes payload contents observable, with repeated-pass stability coverage on the GC-heavy closed-world paths.
- Future enhancement note: value-flow-aware carrier forwarding through locals, `select`, and similar reference-preserving intermediates would be a separate precision project rather than an unfinished blocker for the landed pass.
- `StringGathering` is the principal missing global-post functional parity item.
- `code-pushing`, `code-folding`, `precompute`, `simplify-locals` variants, and `merge-blocks` are placed in intended regions.

## Current Hot Priorities
1. `SimplifyLocals`
   - Main cost center with map-heavy local tracking and repeated rescans.
2. `Vacuum`
   - Most likely contributor to pathological stalls.
   - Full-function reseed and repeated subtree scans are the key cost signal.
3. `AlignmentLowering`
   - Needs module-level/function-level no-op fast paths before full traversal.

## Secondary Risks
- Merge-locals and coalesce-locals cleanups are repeated with limited change-signal gating.
- MergeBlocks and RemoveUnusedBrs likely carry follow-on cost.
- Many correctness-focused passes still use full-module or full-function passes without tight change summaries.

## Measurement Baseline
- Baseline artifacts are maintained via `bun make benchmark-optimize`.
- Useful command set:
  - full pipeline on examples/spec/self-opt artifacts,
  - single-pass microbench (e.g., `simplify-locals`, `vacuum`),
  - spec and user-facing corpus runs.
- `src/passes/merge_blocks_parity_wbtest.mbt` remains the reusable fixed corpus pattern.

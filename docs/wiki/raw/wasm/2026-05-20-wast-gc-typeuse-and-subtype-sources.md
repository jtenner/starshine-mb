## Durable takeaways

- The core type universe is still organized around composite `func`, `struct`, and `array` types, with subtypes optionally declaring supertypes and finality.
- The text format gives type definitions two relevant layers: the type definition itself, and type-use sites such as functions, imports, tags, blocks, `call_indirect`, `return_call_indirect`, and `return_call_ref` that may either name an existing `(type ...)` or spell an inline function signature.
- Recursive groups are semantic grouping constructs, but every member still contributes a flat module type index. Text authors should reason about both source grouping and flat index order.
- Subtyping validation is not a parser-only concern. A source can parse with `(sub ...)`, `final`, and supertype references but still fail validation when the referenced supertype is incompatible, out of range, duplicated, or otherwise violates the validation rules.
- Inline function signatures at type-use sites are abbreviations over function types. A lowering implementation may reuse an equivalent existing function type or append a new one, but it must keep every later `TypeIdx` carrier consistent.

## Starshine implications

- `src/wast/parser.mbt` parses `(type $id ...)`, `(rec ...)`, `(sub final? super* ...)`, custom descriptor metadata, and function type-use syntax with optional `(type idx)` followed by optional inline `(param ...)` / `(result ...)` clauses.
- `src/wast/lower_to_lib.mbt` lowers parsed type definitions into `@lib.RecType` values, resolves named type uses through the `type_ids` map, scans grouped rec members when searching for an equivalent function type, and appends implicit function types after the current flattened type space when no match exists.
- The focused `src/wast/rec_group_typeuse_test.mbt` regression proves one important flat-index invariant: after a two-member rec group, an implicit function type generated by an inline function signature occupies flat type index `2`.
- Starshine docs should keep type-use rules centralized in `docs/wiki/wast/gc-type-authoring.md` and link from call/table/tail-call pages instead of duplicating partial `TypeIdx` guidance in every instruction page.

## Follow-up questions

- Starshine's `wt_resolve_named_or_num_index(...)` currently accepts numeric indices without enforcing the passed upper bound. Keep direct validator evidence in the signoff path for type-use changes instead of relying on lowering alone for out-of-range rejection.
- If future WAST work supports more inline function-type unification or explicit `(type idx)` plus inline signature consistency checks, update this source note and the GC type authoring page with the exact accepted and rejected shapes.

# WAST Resource Declaration Sources (2026-05-19)

This source manifest anchors [`../../wast/resource-declaration-authoring.md`](../../wast/resource-declaration-authoring.md). It was prepared to split table, memory, and global **module-field authoring** from the existing runtime instruction guides and the binary section guide.

## Primary external sources checked

- WebAssembly 3.0 text module source, current as checked on 2026-05-19: <https://webassembly.github.io/spec/core/text/modules.html>. This is the text-syntax anchor for `(table ...)`, `(memory ...)`, `(global ...)`, explicit `(import ...)`, explicit `(export ...)`, inline exports, table element abbreviations, and global initializer expressions.
- WebAssembly 3.0 text type source, current as checked on 2026-05-19: <https://webassembly.github.io/spec/core/text/types.html>. This anchors limits, table types, memory types, global mutability syntax, value types, and reference types as used by resource declarations.
- WebAssembly 3.0 abstract module syntax, current as checked on 2026-05-19: <https://webassembly.github.io/spec/core/syntax/modules.html>. This separates imports from local table, memory, and global definitions and records the module-level resource index spaces.
- WebAssembly 3.0 binary module source, current as checked on 2026-05-19: <https://webassembly.github.io/spec/core/binary/modules.html>. This anchors section ids and the binary separation between import descriptors and table/memory/global definition sections.
- WebAssembly 3.0 module validation source, current as checked on 2026-05-19: <https://webassembly.github.io/spec/core/valid/modules.html>. This anchors validation-context growth for imports, tables, memories, and globals, and the global-initializer constant-expression contract.
- WebAssembly memory64 proposal source, current as checked on 2026-05-19: <https://webassembly.github.io/memory64/core/>. This remains useful context for Starshine's current text-surface limitation: core and binary can represent `i64` limits, but the WAST resource parser lowers declaration limits through the current `i32` path.

## Starshine repository evidence checked

- [`src/wast/parser.mbt`](../../../../src/wast/parser.mbt) parses table fields with ids, inline exports, limits, element types, and optional table element abbreviations; parses memory fields with ids, inline exports, and natural-number limits; parses global fields with ids, inline exports, mutable-or-immutable global types, and initializer instructions; parses explicit import descriptors for table, memory, and global resources; and parses explicit export descriptors for table, memory, and global resources.
- [`src/wast/parser.mbt`](../../../../src/wast/parser.mbt) does not implement inline table, memory, or global import shorthand in the same way it implements function and tag inline imports. Use explicit `(import "m" "n" (table ...))`, `(import "m" "n" (memory ...))`, and `(import "m" "n" (global ...))` fields for current Starshine WAST fixtures.
- [`src/wast/lower_to_lib.mbt`](../../../../src/wast/lower_to_lib.mbt) resolves imported-prefix `TableIdx`, `MemIdx`, and `GlobalIdx` ids, turns inline table/memory/global exports into ordinary export entries, lowers WAST `Limits` through `wt_limits(...)` to `@lib.Limits::i32(...)`, turns WAST table element abbreviations into active element segment definitions, and lowers global initializer instruction arrays into core `Expr` values.
- [`src/wast/module_wast.mbt`](../../../../src/wast/module_wast.mbt) prints table, memory, global, import, and export fields, including inline exports and table element abbreviations, but prints only the current WAST-local `i32` natural-limit spelling.
- [`src/lib/types.mbt`](../../../../src/lib/types.mbt) represents the long-lived core model with imported resources in `ImportSec`, local resources in `TableSec`, `MemSec`, and `GlobalSec`, `TableType(RefType, Limits)`, `MemType(Limits, shared)`, and `GlobalType(ValType, mutable)`. Core limits support both `I32Limits` and `I64Limits`, and memory types carry a shared flag even though the current WAST declaration path does not expose those spellings.
- [`src/binary/decode.mbt`](../../../../src/binary/decode.mbt) and [`src/binary/encode.mbt`](../../../../src/binary/encode.mbt) preserve table, memory, global, and import/export section shapes. The binary memory type supports shared and memory64 limit tags; the WAST authoring page must not imply that current text fixtures cover all binary/core resource shapes.
- [`src/validate/validate.mbt`](../../../../src/validate/validate.mbt) validates imports before local resource sections, validates table reference types and optional core table initializers, validates memory limits and the shared-memory-with-maximum rule, validates globals incrementally under the environment containing imports plus earlier globals, and checks global initializer expressions with the local immutable-`global.get` extension.
- [`src/validate/typecheck.mbt`](../../../../src/validate/typecheck.mbt) owns the runtime stack typing for instructions that use those declarations, including `global.get`, `global.set`, memory instructions, table instructions, and indirect calls.
- [`src/validate/gen_valid.mbt`](../../../../src/validate/gen_valid.mbt) can generate broader valid resource topology than the WAST parser surface, including memory-limit variants and shared-memory-related coverage when enabled.
- [`src/wast/arbitrary.mbt`](../../../../src/wast/arbitrary.mbt) includes representative table, memory, and global module fields for parser/printer coverage, but remains separate from the typed-validity generator.

## Reconciled takeaways

- The binary guide [`../../binary/type-table-memory-global-tag-sections.md`](../../binary/type-table-memory-global-tag-sections.md) remains the right page for section ids, core/binary resource representation, shared-memory validation, optional core table initializers, local `stringrefs`, and whole-module remap checklists.
- The new WAST resource-declaration page should own fixture-facing table, memory, and global module fields: explicit imports, definitions, inline exports, explicit exports, table element abbreviations, global mutability, initializer ordering, imported-prefix indices, and text-surface caveats.
- Important local/core split: Starshine core and binary support `I64Limits` and shared memories, but current WAST table/memory declarations parse natural min/max limits and lower them through `wt_limits(...)` to `I32Limits`; use direct core or binary fixtures for memory64/table64/shared resource-definition evidence until the text surface is widened.
- Important table split: WAST table element abbreviations lower to active element segments, not to the optional core/binary `Table(..., Some(expr))` initializer field documented on the binary resource page.
- Important import split: current Starshine WAST supports explicit resource imports for table/memory/global resources, but not inline import shorthand on `(table ...)`, `(memory ...)`, or `(global ...)` definitions. Separate explicit `(export ...)` fields can export imported resources.

# WAST Memory Instruction Sources (2026-05-19)

This source manifest anchors [`../../wast/memory-instruction-authoring.md`](../../wast/memory-instruction-authoring.md). It was prepared to split the runtime memory-instruction stack/effect contract from the existing memory-argument guide, which owns `align=`, `offset=`, and explicit-memory-index caveats.

## Primary external sources checked

- WebAssembly 3.0 text instruction source, current as checked on 2026-05-19: <https://webassembly.github.io/spec/core/text/instructions.html>. This is the text-syntax anchor for scalar load/store memory arguments, `memory.size`, `memory.grow`, bulk-memory instructions, and folded instruction examples.
- WebAssembly 3.0 abstract syntax instruction source, current as checked on 2026-05-19: <https://webassembly.github.io/spec/core/syntax/instructions.html>. This separates memory arguments (`memarg`) from memory/data/table instruction immediates and gives the core instruction families.
- WebAssembly 3.0 binary instruction source, current as checked on 2026-05-19: <https://webassembly.github.io/spec/core/binary/instructions.html>. This anchors one-byte scalar load/store opcodes, `memory.size` / `memory.grow`, and `0xFC` bulk-memory subcodes for `memory.init`, `data.drop`, `memory.copy`, and `memory.fill`.
- WebAssembly 3.0 validation instruction source, current as checked on 2026-05-19: <https://webassembly.github.io/spec/core/valid/instructions.html>. This is the main external semantic anchor for stack types: scalar loads/stores use the selected memory address type, `memory.size` / `memory.grow` use the selected memory address type, `memory.init` consumes destination address plus `i32` source/length, `memory.copy` uses destination/source address types plus the smaller address type for length, and `memory.fill` consumes destination address, `i32` byte value, and length in the selected memory address type.
- WebAssembly 3.0 binary module source, current as checked on 2026-05-19: <https://webassembly.github.io/spec/core/binary/modules.html>. This anchors the data-count section relationship needed by `memory.init` and `data.drop` validation.
- WebAssembly 3.0 validation module source, current as checked on 2026-05-19: <https://webassembly.github.io/spec/core/valid/modules.html>. This anchors the validation-context and data-count model that Starshine mirrors with a separate `datacnt_requirement` phase.
- WebAssembly memory64 proposal source, current as checked on 2026-05-19: <https://webassembly.github.io/memory64/core/>. This remains useful for explaining why selected memory address width changes stack types even when the instruction spelling is the same.

## Starshine repository evidence checked

- [`src/wast/keywords.mbt`](../../../../src/wast/keywords.mbt) recognizes scalar load/store names, `memory.size`, `memory.grow`, `memory.fill`, `memory.copy`, `memory.init`, and `data.drop`. It currently has no WAST text keywords for atomic memory instructions.
- [`src/wast/parser.mbt`](../../../../src/wast/parser.mbt) parses scalar load/store memory instructions with WAST-local `MemArg` fields, parses zero-operand `memory.size`, unary `memory.grow`, ternary `memory.fill` / `memory.copy`, `memory.init` with a data index plus three operands, and `data.drop` with a data index. Existing parser tests cover these spellings.
- [`src/wast/lower_to_lib.mbt`](../../../../src/wast/lower_to_lib.mbt) lowers scalar loads/stores through `wt_load_store(...)`, lowers `memory.size`, `memory.grow`, and `memory.fill` to `MemIdx(0)`, lowers `memory.copy` to `(MemIdx(0), MemIdx(0))`, resolves the `memory.init` data index and still lowers the memory operand to `MemIdx(0)`, and resolves `data.drop` data indices.
- [`src/wast/module_wast.mbt`](../../../../src/wast/module_wast.mbt) prints scalar load/store instructions with rendered memory arguments, prints `memory.size`, `memory.grow`, `memory.fill`, `memory.copy`, `memory.init <data>`, and `data.drop <data>`, but does not print explicit memory indices for those runtime memory instructions.
- [`src/lib/types.mbt`](../../../../src/lib/types.mbt) represents scalar loads/stores with `MemArg`, `MemorySize(MemIdx)`, `MemoryGrow(MemIdx)`, `MemoryInit(DataIdx, MemIdx)`, `DataDrop(DataIdx)`, `MemoryCopy(MemIdx, MemIdx)`, and `MemoryFill(MemIdx)`.
- [`src/binary/decode.mbt`](../../../../src/binary/decode.mbt) and [`src/binary/encode.mbt`](../../../../src/binary/encode.mbt) preserve memory/data immediates for runtime memory instructions: one-byte `memory.size` / `memory.grow`, scalar load/store memargs, and `0xFC` subcodes `8` through `11` for `memory.init`, `data.drop`, `memory.copy`, and `memory.fill`.
- [`src/validate/typecheck.mbt`](../../../../src/validate/typecheck.mbt) owns memory instruction stack typing: `memarg_check(...)`, `typecheck_load(...)`, `typecheck_store(...)`, `typecheck_memory_size(...)`, `typecheck_memory_grow(...)`, `typecheck_memory_init(...)`, `typecheck_memory_copy(...)`, `typecheck_memory_fill(...)`, and `typecheck_data_drop(...)`. It matches the official address-width model for scalar loads/stores, size/grow, init destination, and mixed-width copy length, but currently types `memory.fill` length as `i32` rather than the selected memory address type.
- [`src/validate/validate.mbt`](../../../../src/validate/validate.mbt) implements the separate `datacnt_requirement` phase for `memory.init` and `data.drop` body use and tracks body-surface facts for memory bulk operations.
- [`src/validate/gen_valid.mbt`](../../../../src/validate/gen_valid.mbt) emits and scans scalar memory and bulk-memory surfaces for generator coverage, including `[FZG]005` scalar widths/nonzero memargs and broader coverage-forced memory/bulk forms.
- [`src/wast/arbitrary.mbt`](../../../../src/wast/arbitrary.mbt) includes representative WAST text for `memory.fill`, `memory.copy`, and `memory.init` in the widened arbitrary prelude, but remains a parser/printer generator rather than a typed-validity oracle.
- [`src/ir/hot_lift.mbt`](../../../../src/ir/hot_lift.mbt), [`src/ir/hot_lower.mbt`](../../../../src/ir/hot_lower.mbt), [`src/ir/hot_flags.mbt`](../../../../src/ir/hot_flags.mbt), and [`src/ir/effects.mbt`](../../../../src/ir/effects.mbt) carry memory instruction lowering/lifting and effect classification used by optimizer passes.

## Reconciled takeaways

- The existing [`../../wast/memory-argument-authoring.md`](../../wast/memory-argument-authoring.md) remains the right page for `offset=`, `align=`, memory32/memory64 address-width selection, and the current WAST nonzero-memory-index gap.
- The new runtime instruction page should own stack shapes, data-count requirements, side effects/traps, pass rewrite checklists, WAST parser/lowerer/printer caveats, and generator/arbitrary coverage boundaries for scalar loads/stores, `memory.size`, `memory.grow`, `memory.fill`, `memory.copy`, `memory.init`, and `data.drop`.
- Important local/spec divergence to keep visible: official validation types `memory.fill` length with the selected memory address type, while current Starshine `typecheck_memory_fill(...)` requires `i32` length even for memory64. This should be treated as a validator follow-up, not as the intended long-term contract.
- Atomic memory instructions are represented in core/binary/typecheck/generator surfaces, but current WAST keyword/parser evidence does not expose them as text syntax. `[FZG]017` should therefore link to the runtime memory page for binary/generator facts and continue to avoid overclaiming WAST text support until an atomic-focused page or parser work lands.

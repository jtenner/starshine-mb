# WAST String Instruction Sources (2026-05-19)

This source manifest anchors [`../../wast/string-instruction-authoring.md`](../../wast/string-instruction-authoring.md). It was prepared to split the string instruction stack/type contract from the older [`../../strings/string-const-surface.md`](../../strings/string-const-surface.md), which remains focused on literal identity and Starshine's local `StringRefsSec` binary pool.

## Primary external sources checked

- WebAssembly stringref proposal overview, current as checked on 2026-05-19: <https://github.com/WebAssembly/stringref/blob/main/proposals/stringref/Overview.md>. This is the primary proposal-facing semantic anchor for `stringref`, `string.const`, array-backed string construction, and array-backed encoding operations. It also records many string operations that Starshine does **not** currently model.
- WebAssembly proposals index / stringref repository context, current as checked on 2026-05-19: <https://github.com/WebAssembly/stringref>. This keeps the string instruction page explicit that the surface is proposal-facing rather than part of the stable core WebAssembly 3.0 section model used by the binary resource pages.
- WebAssembly 3.0 text instruction source, current as checked on 2026-05-19: <https://webassembly.github.io/spec/core/text/instructions.html>. This is useful mainly as a negative/current-core anchor: the stable 3.0 text instruction page does not define the Starshine string helper spellings described here.
- WebAssembly 3.0 validation instruction source, current as checked on 2026-05-19: <https://webassembly.github.io/spec/core/valid/instructions.html>. This remains the stable-core stack-typing reference for ordinary instructions, while the stringref proposal and Starshine validator evidence own the string-helper stack rules below.
- Prior wiki source snapshot [`2026-05-13-type-table-memory-global-tag-sources.md`](2026-05-13-type-table-memory-global-tag-sources.md), which checked the WebAssembly 3.0 and js-string-builtins module sources and found no stable core `stringrefs` section id. Keep that caveat attached to Starshine's local section id `14`.

## Starshine repository evidence checked

- [`src/wast/keywords.mbt`](../../../../src/wast/keywords.mbt) recognizes `string.const`, `string.new_utf8_array`, `string.new_wtf16_array`, `string.encode_utf8_array`, `string.encode_wtf16_array`, `string.new_lossy_utf8_array`, `string.new_wtf8_array`, `string.encode_lossy_utf8_array`, and `string.encode_wtf8_array` as WAST opcodes.
- [`src/wast/parser.mbt`](../../../../src/wast/parser.mbt) parses `string.const` with one text literal decoded to bytes, and parses the eight array new/encode helper names as no-immediate instructions. It currently does not expose the wider proposal families such as measurement, comparison, hash, view, iterator, or `string.new_utf8` memory-buffer forms.
- [`src/wast/lower_to_lib.mbt`](../../../../src/wast/lower_to_lib.mbt) lowers the WAST string opcodes into the matching [`src/lib/types.mbt`](../../../../src/lib/types.mbt) `Instruction` variants. `string.const` requires a literal in the WAST opcode path and becomes `Instruction::string_const(bytes)` after parsing. Existing lowerer tests cover string array instructions, `stringref` text types, and `string.const` literals.
- [`src/wast/module_wast.mbt`](../../../../src/wast/module_wast.mbt) prints the same supported string opcode set, preserving string literal bytes through `quote_wast_bytes(...)` for `string.const`.
- [`src/lib/types.mbt`](../../../../src/lib/types.mbt) represents `StringConst(Bytes)`, the eight string array new/encode helpers, and `ValType::stringref()` / `AbsHeapType::StringAbsHeapType`.
- [`src/binary/decode.mbt`](../../../../src/binary/decode.mbt) and [`src/binary/encode.mbt`](../../../../src/binary/encode.mbt) use the local `0xFB` string instruction subcodes currently implemented by Starshine: `0x82` for `string.const`, `0xB0` through `0xB7` for the array new/encode helpers. Encoding `string.const` requires module string-pool context; decoding resolves the literal index through the local `StringRefsSec` before later consumers see the instruction.
- [`src/validate/typecheck.mbt`](../../../../src/validate/typecheck.mbt) types `string.const` as producing `stringref`; types string new-array helpers as consuming an array reference plus `i32` start and `i32` length and producing `stringref`; and types encode-array helpers as consuming `stringref`, a mutable matching array reference, and an `i32` start before producing an `i32` result.
- [`src/validate/gen_valid.mbt`](../../../../src/validate/gen_valid.mbt) emits supported string operations in coverage-forced valid modules when the generated GC type plan contains matching `i8` or `i16` array types, and scans the instruction tree to set `has_string_ops` for `StringOps` coverage.
- [`src/validate/validate.mbt`](../../../../src/validate/validate.mbt) exposes the `StringOps` coverage key and the coverage-forced test that requires nonzero string-operation coverage alongside GC constructor/accessor coverage.
- [`src/wast/arbitrary.mbt`](../../../../src/wast/arbitrary.mbt) currently has no dedicated string instruction emission path. WAST arbitrary output should therefore not be treated as string-helper text coverage until a parser/printer roundtrip slice is added.

## Reconciled takeaways

- The new WAST string instruction page should own text spellings, stack shapes, current proposal/local caveats, array mutability/storage requirements, generator/arbitrary coverage boundaries, and pass rewrite guidance for supported string helper instructions.
- [`../../strings/string-const-surface.md`](../../strings/string-const-surface.md) remains the durable page for `string.const` literal identity, the local `StringRefsSec` pool, binary value-type decoder caveats, and the section-id-`14` local-extension warning.
- Starshine's implemented WAST string helpers are narrower than the stringref proposal. Measurement, comparison, hash, view/iterator, memory-buffer new/encode, and non-array helper families need new AST, binary, validator, generator, and WAST evidence before wiki pages or tests should claim support.
- Array-backed string helpers depend on GC array types: UTF-8/WTF-8 helpers require `i8` array storage, WTF-16 helpers require `i16` array storage, and encode helpers require a mutable destination array. Pair this page with [`../../wast/gc-type-authoring.md`](../../wast/gc-type-authoring.md) and [`../../wast/gc-aggregate-instruction-authoring.md`](../../wast/gc-aggregate-instruction-authoring.md) when authoring the array types and setup instructions.

# Starshine Release Package Surface Refresh - 2026-06-04

This manifest refreshes the release-process source boundary after the repo moved to the new `moon.mod` module-file format and after the Node package surface was rechecked against its explicit export map. It supersedes the stale local-file references in [`2026-05-20-starshine-release-process-sources.md`](2026-05-20-starshine-release-process-sources.md) while preserving that earlier manifest as provenance for the original npm packaging conclusions.

## External primary sources checked

1. npm CLI v11 `package.json` docs: <https://docs.npmjs.com/cli/v11/configuring-npm/package-json>
   - Rechecked `name` / `version` as the publish identity, `files` as the tarball inclusion allow-list, `exports` as the public entry-point boundary, `bin` as CLI-link metadata, `engines` as Node-version metadata, and `private` as accidental-publication protection for the root script package.
2. npm CLI v11 `npm pack` docs: <https://docs.npmjs.com/cli/v11/commands/npm-pack>
   - Rechecked `npm pack --dry-run` as the local tarball-inspection path and confirmed that `dry-run` is an npm command option for pack/publish-style operations.
3. npm CLI v11 `npm publish` docs: <https://docs.npmjs.com/cli/v11/commands/npm-publish>
   - Rechecked that publish writes a package to the configured registry, that a name/version tuple cannot be reused after publication, that included files should be inspected with `npm pack --dry-run`, and that actual publish remains a human-controlled registry-write operation.
4. npm CLI v11 lifecycle script docs: <https://docs.npmjs.com/cli/v11/using-npm/scripts#life-cycle-scripts>
   - Rechecked that `prepack` runs before `npm pack` and `npm publish`; this matters because Starshine's Node package uses `prepack: npm run build`.
5. MoonBit v0.9.3 module configuration docs: <https://docs.moonbitlang.com/en/latest/toolchain/moon/module.html>
   - Rechecked that `moon.mod` is the new module-file format, `moon.mod.json` is legacy, and `version` is the module version used for MoonBit package publication.

## Starshine local sources checked

- [`AGENTS.md`](../../../../AGENTS.md) and [`docs/README.md`](../../../README.md) still require explicit semver bumping, successful validation, consistent package version bumps, release notes from wiki/log/git evidence, and human-controlled publication.
- [`moon.mod`](../../../../moon.mod) is the current MoonBit metadata file. It declares `name = "jtenner/starshine"` and `version = "0.1.0"`; the older `moon.mod.json` path in the 2026-05-20 release manifest is stale.
- [`node/package.json`](../../../../node/package.json) declares the npm package `@jtenner/starshine` at `0.1.0`, ESM package mode, explicit export subpaths, `bin.starshine`, `files`, `engines.node >=25`, and `prepack: npm run build`.
- Root [`package.json`](../../../../package.json) remains a private script package (`"private": true`) named `starshine-scripts`; it is not a published Starshine artifact and should not be version-synchronized with product releases.
- [`node/README.md`](../../../../node/README.md) describes the public Node subpaths and the build boundary: `npm run build` refreshes the WASI CLI artifact, while wrappers and the wasm-gc adapter artifact are not regenerated by that build.
- [`node/internal/.gitignore`](../../../../node/internal/.gitignore) ignores both wasm package artifacts locally, while [`node/internal/.npmignore`](../../../../node/internal/.npmignore) explicitly keeps those wasm artifacts publishable even though they are gitignored.
- [`scripts/lib/build-node-package.mjs`](../../../../scripts/lib/build-node-package.mjs) builds `src/cmd` for `wasm --release`, copies `_build/wasm/release/build/cmd/cmd.wasm` to `node/internal/starshine.wasm-wasi.wasm`, and requires `node/internal/starshine.wasm-gc.wasm` to already exist. The script comment still calls the wasm-gc file a checked-in boundary artifact, but current Git metadata shows it is an ignored local/package artifact rather than tracked source.
- [`scripts/lib/generate-node-package.mjs`](../../../../scripts/lib/generate-node-package.mjs) intentionally throws, so wrapper-generation is not part of release build automation today.
- [`scripts/lib/validate-task.ts`](../../../../scripts/lib/validate-task.ts) remains the source of repo validation command shape; the release page should link to the validation-gates page rather than duplicating the whole gate implementation.

## Durable conclusions

1. A Starshine product release currently has **two versioned product metadata surfaces**: MoonBit [`moon.mod`](../../../../moon.mod) and npm [`node/package.json`](../../../../node/package.json). They should be bumped together for a normal published Starshine release.
2. The root [`package.json`](../../../../package.json) is a private script-orchestration package and is not part of the public version-synchronization rule.
3. Node publication is bounded by `node/package.json#exports`, `files`, `bin`, lifecycle scripts, and the tarball produced from the `node/` package directory. A release candidate must inspect `npm pack --dry-run` output from `node/`, not infer package contents from the repo tree.
4. `prepack: npm run build` means `npm pack` / `npm publish` will rebuild the WASI CLI artifact unless scripts are suppressed. That build does **not** regenerate JS/TS wrappers or the wasm-gc adapter artifact, so wrapper/API drift must be reviewed and tested as checked-in source.
5. `node/internal/starshine.wasm-gc.wasm` is a required local/package artifact for the current build, but it is ignored by Git and explicitly unignored for npm packaging by `node/internal/.npmignore`. Future release work should inspect both local artifact presence and tarball contents instead of calling it checked-in evidence.
6. Registry writes, npm tokens, public tags, and remote pushes remain human-controlled operations even when an agent prepares release docs, version bumps, or validation summaries.

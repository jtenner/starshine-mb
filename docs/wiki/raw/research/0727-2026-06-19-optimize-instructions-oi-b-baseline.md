## Direct compare classification

Agent judgment: the 27 raw mismatches are **parity gaps**, not semantic-safe accepted drift and not Starshine wins. The inspected failure dirs are all generated by `gen_valid` and share the same generated template; Binaryen applies broader `OptimizeInstructions.cpp` canonicalization that current Starshine intentionally has not implemented yet.

Recurring observed Binaryen-side transforms in all 27 mismatching WAT diffs:

- integer constant subtraction folding, e.g. `i32.sub (i32.const 84) (i32.const 3)` to `i32.const 81`;
- integer constant subtraction folding for i64, e.g. `i64.sub (i64.const 99) (i64.const 5)` to `i64.const 94`;
- floating subtraction canonicalization to add-negative constants, e.g. `f32.sub x 2.5` to `f32.add x -2.5` and `f64.sub x 4.5` to `f64.add x -4.5`;
- `i32.wrap_i64` of a constant to an `i32.const`;
- floating divide-by-two spelling canonicalization to multiply-by-half in the generated template.

These are not evidence of invalid Starshine output: `validationFailureCount` was `0`. They are also not accepted as harmless output shape drift because the current OI audit requires Binaryen behavior-parity evidence or explicit Starshine wins. The correct owner is the later scalar/default arithmetic breadth work, especially `[O4Z-AUDIT-OI-D]`, with any floating behavior considered carefully under trap/NaN/fast-math constraints before implementation.

The single command failure is classified as a **tool/Binaryen failure**: `case-000029-wasm-smith` failed Binaryen/canonicalization with `Recursion groups of size zero not supported`, while Starshine produced `starshine.raw.wasm`. This is not a Starshine semantic mismatch.

Because the 1000-case lane hit the default failure ceiling after only 54 compared cases, the requested `10000` follow-up was not run; the lane was not stable enough for widening under the OI-B instructions.

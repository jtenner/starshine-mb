# HSO `try_table` swap micro-audit

## Question

Can the `trySwap(...)` wrapper/effect part of HSO-G be narrowed for `try_table` roots without approving all remaining swap operand/effect drift?

## Scope audited

This note covers explicit `try_table` roots and result-typed `try_table` wrappers already probed against Binaryen `version_130` or generated by the dedicated HSO profile:

1. direct void `try_table` roots between a resource-reading constructor operand and a later `struct.set`;
2. non-throwing unrelated-global-set `try_table` roots that Binaryen crosses;
3. catch-taken or catchable-call `try_table` roots that Binaryen treats as no-fold boundaries;
4. descriptor constructor siblings for non-throwing and catchable `try_table` roots;
5. result-typed wrappers with cross-family stores/growths, same-resource barriers, and catchable calls; and
6. generated profile roots that exercise these families under the dedicated HSO lane.

It does **not** close non-`try_table` wrappers, arbitrary `trySwap(...)` operands, exact descriptor `ref.cast`, direct/indirect/typed call constructor operand boundaries outside the listed roots, or performance/open output-shape items.

## Source-backed behavior map

| Family | Evidence | Current classification |
| --- | --- | --- |
| Direct void `try_table` with cross-family ordinary stores | `0910`, `0911` | Boundary for the probed direct wrapper shape: Binaryen preserves the resource read, wrapper, store, and later `struct.set`; Starshine matches. |
| Direct void `try_table` with same-resource fills | `0912` | Same-effect boundary: Binaryen and Starshine preserve the later `struct.set`. |
| Direct void `try_table` with cross-family growth roots | `0913` | Direct wrapper boundary for these probed growth roots; does not supersede direct non-`try_table` cross-family growth folds from `0888`. |
| Block-wrapped non-throwing `try_table` / unrelated `global.set` | `0927`, `0979`, `0980` | Parity covered positive: Binaryen folds resource-reading constructors through non-throwing unrelated global writes while preserving needed wrappers; Starshine now matches. |
| Catch-taken `throw` and catchable direct call inside `try_table` | `0981`, `0982` | Parity covered negative: local catchability can skip the constructor assignment, so the later `struct.set` remains. |
| Descriptor constructor non-throwing/catchable split | `0983`, `0984` | Parity covered: immutable/pure descriptor non-throwing roots fold; catchable-call descriptor roots preserve `struct.set`. |
| Result-typed non-throwing unrelated-global roots | `0985`, `0987` | Parity covered positives for memory/table resource reads; Starshine preserves catch-label wrappers instead of invalid peeling. |
| Result-typed same-resource and catchable-call barriers | `0986`, `0988`, `0991`-`0994`, `0997`, `0998` | Parity covered negatives: same-resource effects and catchable direct/indirect/typed-function-reference calls keep the later `struct.set`. |
| Result-typed cross-family store/growth roots | `0989`, `0990`, `0995` | Parity covered for the source-backed positive/negative split: pure cross-family stores/growths fold; catchable typed-call cross-store roots stay blocked. |
| Descriptor result-typed direct/indirect/typed call set-value folds | `1005`, `1007`, `1008` | Parity covered positives for pure immutable descriptor operands; mutable-descriptor/callable old-field siblings stay separate and are not generalized here. |
| Dedicated generated `try_table` roots | `1024`-`1026`, `1034`-`1039`, `1042`-`1044`, refreshed `1073` | Generated coverage exercises void and result-typed `try_table` positives and barriers. The `1073` aggregate lane is green with the documented local-cleanup normalizer. |

## Narrow closure

The explicit `try_table` swap/wrapper matrix above is narrow-closed for HSO-G:

- positives and negatives are source-backed rather than inferred from validation-only lanes;
- Starshine either has focused tests or dedicated-profile roots for each listed family;
- the earlier `0922` non-throwing `try_table` / unrelated-global parity gap was fixed in `0927`, and result-wrapper peeling drift was fixed in `0985` rather than accepted; and
- no broad `trySwap(...)` drift is hidden behind this closure.

HSO-G remains open for broader swap operands/effects, non-`try_table` wrapper peeling or flattening, call-valued constructor operand boundaries outside the mapped families, exact descriptor `ref.cast`, old-field trap-preservation variants not already covered by their own notes, and performance/final closeout.

## Validation

Docs/status-only micro-audit. No new code changed and no tests were required for this note. Supporting focused tests, generated profile tests, direct smokes, and the `1073` dedicated-profile lane are cited above.

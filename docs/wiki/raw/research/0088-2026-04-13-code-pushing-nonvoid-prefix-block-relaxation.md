# 0088 - `code-pushing` candidate-aware non-void explicit-exit block-prefix relaxation

## Context

- Follow-up to [`0080`](./0080-2026-04-12-code-pushing-crossed-condition-set-alias.md), [`0081`](./0081-2026-04-12-code-pushing-explicit-exit-fed-tail.md), and [`0087`](./0087-2026-04-13-code-pushing-standalone-func1977-hot-lower-recursive-fix.md).
- After `0087`, the reduced standalone `Func 1977` writeback failure was closed, but current-source standalone replays still showed remaining pass-level drift around the earlier decode ladders in standalone `Func 1977` and the repeated sibling family in standalone `Func 1948`.
- A first broad relaxation of the non-void explicit-exit prefix fence made the intended standalone HOT regions move again, but it also reopened some older reduced negatives. Those negatives needed to be rechecked against Binaryen before keeping or reverting the change.

## Problem Restatement

The remaining current-tree pass mismatch was narrower than the old blanket non-void fence:

- Binaryen still moves later `local.set` roots past an `if` even when an earlier prefix `Block` has explicit exits,
- but only when that earlier block prefix is not the real same-source crossed-condition-set carrier hazard.

The old non-void-region prefix guard was too coarse because it rejected every unsafe explicit-exit prefix root before the pushpoint, even when the only risky prefix root was an earlier block that Binaryen still moves past in practice.

## Reduced Evidence

### 1. Earlier branch-root and branch-free crossed-gap negatives were stale

Direct Binaryen probes on reduced fixtures now show that Binaryen **does** reorder these families:

- `probe-branch-root-branch-free-a.wat`
  - Binaryen moves the later `local.set $3 (i32.const 7)` after the later `if`.
- `probe-zero-result-branch-root-a.wat`
  - Binaryen moves the later `local.set $2 (i32.const 7)` after the later `if`.
- `probe-nested-branch-body-a.wat`
  - Binaryen moves the later `local.set $2 (i32.const 7)` after the later `if` even though the earlier branch only appears inside a nested block body region.
- `probe-parent-escape-branch-free-a.wat`
  - Binaryen moves the later alias `local.set $5 (local.get $3)` after the later `if` even with the earlier parent-escape carrier scaffold.

So those old reduced negatives were no longer valid pass fences.

### 2. The real same-source crossed-condition-set case is still negative

`probe-same-source-crossed-a.wat` still shows no Binaryen `--code-pushing` diff.

That means the kept blocker is still the true same-source alias family:

- an earlier alias `local.set` of the same carried source local
- whose written local feeds the target `if` condition
- while another same-source alias root is still being considered for motion

### 3. Current-source standalone replay moved the intended current-tree families again

Using current-source `moon run src/cmd --target native -- --code-pushing ...` on the standalone reconstructions:

- standalone `Func 1977` again admits the reopened local-`38`, local-`33`, and local-`45` motion family at the pass level
- standalone `Func 1948` again admits the repeated local-`135`-through-`145` and local-`153`-through-`163` family at the pass level
- the remaining exact standalone Starshine-vs-Binaryen WAT drift is therefore narrower than another missed pushpoint-admission blocker and is mixed with the already-known no-pass / writeback canonicalization noise in those autogenerated standalone modules

## Landed Fix

In [`src/passes/code_pushing.mbt`](../../../../../src/passes/code_pushing.mbt):

1. `cp_try_push_to_pushpoint(...)` no longer treats every unsafe explicit-exit prefix root in a non-void region as an unconditional blocker.
2. The non-void prefix decision is now candidate-aware for pushpoint motion:
   - earlier unsafe `Block` prefixes can be tolerated for the currently considered candidate when that is what the Binaryen-matched reduced probes require.
3. Two same-source guards keep the real hazard fenced:
   - the existing crossed-condition-set alias guard for the obvious crossed-gap case
   - a new prior/later same-source alias-condition-set check so the pass does **not** reorder the real reduced same-source carrier family

## Durable Conclusion

The remaining post-`0087` pass drift was not another blanket non-void explicit-exit fence question. The right rule is narrower:

- earlier unsafe block prefixes are **not** inherently blockers for later non-void-region pushpoint motion
- the real blocker is still the same-source alias-condition-set carrier family

That candidate-aware relaxation reopens the Binaryen-matched reduced branch-root and parent-escape crossed-gap cases, keeps the true same-source crossed-condition-set family fenced, and restores the intended current-source standalone `Func 1948` / `Func 1977` pass-level moves.

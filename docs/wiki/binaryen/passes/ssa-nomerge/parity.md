# `ssa-nomerge` Binaryen Parity

This page is the current local signoff tracker.
Use the strategy and shape pages in this folder for the upstream Binaryen algorithm itself; use `[./implementation-structure-and-tests.md](./implementation-structure-and-tests.md)` for owner-file and proof-surface navigation. This page is about the repo's current evidence and remaining gaps.

## Durable Conclusions

- The June 2026 exceptional-edge corruption family is fixed in-tree by making HOT `ssa-nomerge` fail closed on `try_table` / throw-family flow, matching the documented SSA v1 normal-flow-only contract.
- The June 2026 default-local branchy-control drift family is fixed in the raw no-write path: reads of never-written body locals are materialized as explicit type defaults across block/loop/if/try_table bodies without invoking HOT SSA.
- The June 2026 final-lane dropped-unreachable-debris drift family is fixed in the same no-write raw path: `drop (unreachable)` / nested dropped-unreachable debris before a hard `unreachable` is collapsed using the existing shared cleanup helper.
- The April `2026-04-10` dead-param-write parity family is now fixed in-tree for `ssa-nomerge`.
- Current source-mode `ssa-nomerge` replay on the checked-in debug CLI artifact completes and validates, but exact normalized/canonical artifact equality is not currently claimed; the latest direct replay still differs at `defined=0 abs=27`.
- Random compare fuzz is still useful, but it is not sufficient as the only signoff lane for `ssa-nomerge`.
- The 2026-05-07 current-head local-declaration shaping drift family is fixed in-tree: `ssa_destroy` now matches Binaryen temp-local preservation/reuse for unreachable dead `local.tee` value-if carriers without generic lowerer trimming.
- Fresh post-merge reruns on seed `0x51a` stayed clean in both the mixed-generator and `gen-valid` lanes.
- The reduced unreachable compare-carrier slice behind the traced `Func 523` family is now covered in-tree in both lift and pass tests.
- The old traced `Func 523` `writeback-validate:type mismatch` skip is now retired by the focused extracted-function CLI replay test.
- Exact raw-byte parity is still not claimed because Starshine's raw lowering path can still fail closed on artifact-only writeback families, and this follow-up did not rerun a fresh full-artifact traced skip census.
- The non-current subtype/result block-exit edge clusters now have source-backed coverage for `br_on_non_null` tested-payload widening (`ref.i31` to non-null `eqref`, child-to-parent struct/array producers, and child-to-parent `ref.func` producers) and for `br_on_null` prefix-payload widening (`ref.i31` to nullable `eqref`, child-to-parent struct/array producers, and child-to-parent `ref.func` producers); no behavior-code widening was needed beyond the existing raw branch copy lowering, spill helper, and reference matcher.

## Remaining Gap

- Dropped-local-increment debris no longer triggers a whole-function HOT fail-closed skip. HOT SSA rewrite preserves reaching locals for the exact `drop(i32.add(local.get, i32.const 1))` shape and `local.tee` variants on reduced HOT-path fixtures; reversed operands and non-`1` constants remain validated boundary shapes. Broader integer widths remain unclaimed until tested.
- A tiny nested-CFG subset is now supported on the HOT path: straight-line `drop(block (result t) ...)` value blocks with only local get/set/tee, constants, and simple numeric ops (`ssa_simple_drop_value_block_allowed(...)` plus inline rename in `ssa_local_visit_node`), and straight-line `drop(if (result t) ...)` with a single scalar numeric result (`i32`/`i64`/`f32`/`f64`), both arms present, and a straight-line terminator-free condition with local get/set/tee and simple numeric traffic, or one simple `block (result i32)` condition with the same straight-line local traffic (`ssa_simple_condition_value_block_allowed(...)` / `ssa_simple_result_if_condition_allowed(...)` / `ssa_simple_drop_result_if_allowed(...)` plus CFG-backed rename/use-def inline visits with per-root segment isolation and join phis on the post-`drop` continuation). Condition-local writes are allowed only because CFG keeps branch-free condition blocks in the predecessor block that dominates both arms and rename/use-def visit that block before the arm regions; branches and nested control in the condition stay fail-closed. The dropped result value is not SSA-modeled; arms still need CFG/SSA for local traffic. `v128`, reference/GC, and multi-value result-if remain unsupported. `ssa_nomerge_has_nested_cfg_sensitive_control(...)` still fail-closes all other nested value-control shapes.
- Nested CFG-sensitive behavior remains fail-closed for HOT SSA mutation beyond the supported subsets and is regression-covered: nested result-typed `if` inside dropped value blocks, nested `br` / `br_if` / `loop` / nested `if` inside supported operand shapes, missing-else or multi-value operand `if`, and broader nested control must not produce `pass[ssa-nomerge]:mutated` on the hot path and must leave the hot-path revision unchanged. Root-level value blocks and structured control bodies whose children are region-root slots remain outside this guard and may still rewrite through HOT SSA or the raw no-write path.
- Future work requires nested-region-aware liveness/use-def for unsupported shapes, exact local read discovery inside arbitrary nested value control, rename/destruction for nested `if`/loop/branch shapes, and lowering tests that prove local copies are inserted only at valid stack-neutral sites — not Binaryen parity claims for full nested CFG-sensitive SSA.
- Void loop backedges now have a narrow raw structured subset: void `loop` bodies and void loop-target `br` / `br_table` backedges can rewrite without entering HOT SSA, using loop-header read-before-write facts to preserve loop-carried canonical locals and to freshen only dead loop-local writes. This subset covers the source-backed no-copy loop-target `br_table` and loop-carried read-before-write `br_table` regressions; no-copy loop-target `br_if` is also admitted by the later conditional-backedge slice. Copy-needing conditional loop backedges, result-carrying loop targets, and broader loop-param ABI remain unsupported.
- Typed loop-param control has narrow raw structured exceptions for direct current-loop backedges. No-result single-param loops use the proxy value-block model for scalar and reference branch families; no-result multi-param loops use a scratch-per-param store model for direct `br`, `br_if`, all-current-target `br_table`, and no-copy `br_on_null` whose tested nullable ref is a lowered `local.get`; single-result loops use the same store model plus a synthetic outer result block for direct `br`, `br_if`, all-current-target `br_table`, and the same no-copy `br_on_null` subset. Multi-result typed loops, non-current-loop `br_table` targets, branch-bearing nested `loop` / `try_table` bodies, copy-needing multi-param `br_on_null`, arbitrary-tested `br_on_null`, and broader result-loop/reference `br_on_*` header models remain explicitly unsupported and intentionally fail-closed. HOT represents typed loop params as `BlockResultResolved` param prefixes on `HotOp::Loop` children and as branch operands on backedges, with `HotLabelInfo.branch_arity` recording the payload count. SSA v1's local overlay models only `LocalGet` / `LocalSet` / `LocalTee` reaching definitions; it does not treat loop-param stack flow as local defs, and destruction only inserts predecessor copies for overlay phis on normal CFG edges — not wasm branch operands that re-enter typed loop headers.
- `ssa-nomerge` is safe because typed-loop-param HOT mutation still skips via `ssa_nomerge_has_loop_param_control(...)`; the raw loop-target exceptions are separate, limited to direct current-loop backedges recognized before rewrite, and use stack-neutral or edge-local copy placement before the proxy/store lowering updates loop-param scratches.
- Broader typed-loop-param behavior is fail-closed for HOT SSA mutation and regression-covered: pipeline traces must not contain `pass[ssa-nomerge]:mutated` for unsupported typed-loop-param fixtures, locals written before the loop stay preserved, and backedge-carried param values stay separate from unrelated body locals. The older `br_if` fixture-validation blocker is superseded by the current scalar WAT and concrete-ref programmatic mixed-family fixtures; remaining fail-closed cases are multi-result typed loops, non-current `br_table` targets, branch-bearing nested `loop` / `try_table`, and broader loop-param/header SSA shapes.
- Future work requires loop-param-aware liveness, use-def facts for branch operands, phi placement at typed loop headers, rename semantics that distinguish entry vs backedge param values, destruction copies before loop backedge terminators, lowering that preserves typed loop branch operands for the still-unsupported branch/result shapes, and validated coverage for each widened subset — not broad Binaryen parity claims for typed-loop SSA.
- Exceptional-edge SSA remains explicitly unimplemented and intentionally deferred. The CFG already records `ExceptionalEdge` successors for `try` / `try_table` headers and throw-family terminators, but SSA v1 liveness, dominators, phi placement, dominator-stack rename, and predecessor-copy destruction all follow the normal-flow-only contract.
- Catch continuations are not dominated by throwing blocks.

- `ssa-nomerge` is safe because it skips exceptional-flow HOT mutation via `ssa_nomerge_has_exceptional_flow(...)`, not because the local SSA overlay models exceptional successors.
- Exceptional-flow behavior is fail-closed for HOT SSA mutation and regression-covered: `try_table` write-before-throw, default-local catch continuation, conditional throw with a preceding local write, legacy `rethrow` / `delegate` shapes, and a pipeline trace that must not contain `pass[ssa-nomerge]:mutated`. These tests prove preserved local traffic where writes exist; they do not enable exceptional SSA rewriting.

# `optimize-instructions` fuzzing and parity sweep

Recommended smoke lane: run the ordinary GenValid compare-pass lane for this pass:

```sh
bun scripts/pass-fuzz-compare.ts --count 10000 --seed 0x5eed --pass optimize-instructions --out-dir .tmp/pass-fuzz-optimize-instructions --jobs auto --starshine-bin target/native/release/build/cmd/cmd.exe
```

## Family-based sweep

Use the OI parity matrix and sweep wrapper before adding another one-off behavior probe:

```sh
bun scripts/oi-parity-sweep.ts --priority P0
```

Then execute a bounded family with the default GenValid generator before the dedicated profile exists:

```sh
moon build --target native --release src/cmd
bun scripts/oi-parity-sweep.ts \
  --family OI-D \
  --default-gen-valid \
  --execute \
  --count 100 \
  --starshine-bin target/native/release/build/cmd/cmd.exe
```

The dedicated row profiles now exist as smoke-level GenValid profiles, so the ordinary profile-backed row no longer needs `--include-profile-stubs`:

```sh
bun scripts/oi-parity-sweep.ts \
  --family OI-D \
  --execute \
  --count 100 \
  --starshine-bin target/native/release/build/cmd/cmd.exe
```

The wrapper calls `scripts/pass-fuzz-compare.ts --pass optimize-instructions`, which compares Binaryen `wasm-opt --all-features --optimize-instructions` against Starshine `--optimize-instructions` through the existing canonicalized normalized-WAT comparison path. Executed sweeps now print a `Result summaries` section with per-row match/mismatch/failure counts, `genValidProfileCaseCounts`, and `cases.jsonl` status groups by `genValidProfileCaseLabel`; when profile labels are absent, transformed GenValid cases fall back to `transform:<id>` groups. Pass `--summarize-existing` with the same filters and `--out-dir` to re-read existing sweep output without rerunning comparisons.

## Case-label classification convention

For trigger-profile and transform rows, classify the grouped summary before opening another one-off probe. Use `genValidProfileCaseLabel` or the fallback `transform:<id>` label as the grouping key, inspect one representative `binaryen.wat` / `starshine.wat` diff per label, and record the result in the row's `classifiedCaseLabels` array in [`parity-matrix.json`](./parity-matrix.json). Each entry records the label, transform id, representative failure directory, matrix status, agent classification, observed diff, next action, and evidence. The status is an agent judgment; the harness only reports mismatch mechanics.

## Dedicated GenValid smoke profiles

The matrix records the first pass-specific profiles and `src/validate/gen_valid.mbt` accepts them. Most remain smoke-level configs: each maps to a bounded Binaryen-oracle GenValid recipe with the right broad feature floor, but does not yet guarantee an OI-specific trigger in every generated module. `pass-oi-default-scalar`, `pass-oi-local-facts`, `pass-oi-memory-bulk`, and `pass-oi-ref-gc` are now stronger than broad configs: they emit trigger smoke modules for OI-D/OI-E/OI-G/OI-I. OI-D/OI-E/OI-I still use three-way selectors; OI-G now uses a seven-way selector covering direct tiny bulk, memory64 dynamic/local-carried bulk, live private-global/dead-trap boundaries, no-param call/existing-producer wrapped bulk, narrow-store masks, dynamic i32 local.tee bulk operands, and multi-memory bulk operations. The GenValid manifest records each selected OI trigger as `profile_case_label`, and compare-pass copies that into `result.json` `genValidProfileCaseCounts`, `summary.json` strategy counters, and `cases.jsonl` `genValidProfileCaseLabel` fields for mechanical family triage.

# `optimize-instructions` fuzzing and parity sweep

Recommended smoke lane: run the ordinary GenValid compare-pass lane for this pass:

```sh
bun scripts/pass-fuzz-compare.ts --count 10000 --seed 0x5eed --pass optimize-instructions --out-dir .tmp/pass-fuzz-optimize-instructions --jobs auto --starshine-bin target/native/release/build/cmd/cmd.exe
```

## Family-based sweep

Use the OI parity matrix and sweep wrapper before adding another one-off behavior probe:

```sh
bun scripts/oi-parity-sweep.ts --priority P0
```

Then execute a bounded family with the default GenValid generator before the dedicated profile exists:

```sh
moon build --target native --release src/cmd
bun scripts/oi-parity-sweep.ts \
  --family OI-D \
  --default-gen-valid \
  --execute \
  --count 100 \
  --starshine-bin target/native/release/build/cmd/cmd.exe
```

The dedicated row profiles now exist as smoke-level GenValid profiles, so the ordinary profile-backed row no longer needs `--include-profile-stubs`:

```sh
bun scripts/oi-parity-sweep.ts \
  --family OI-D \
  --execute \
  --count 100 \
  --starshine-bin target/native/release/build/cmd/cmd.exe
```

The wrapper calls `scripts/pass-fuzz-compare.ts --pass optimize-instructions`, which compares Binaryen `wasm-opt --all-features --optimize-instructions` against Starshine `--optimize-instructions` through the existing canonicalized normalized-WAT comparison path. Executed sweeps now print a `Result summaries` section with per-row match/mismatch/failure counts, `genValidProfileCaseCounts`, and `cases.jsonl` status groups by `genValidProfileCaseLabel`; when profile labels are absent, transformed GenValid cases fall back to `transform:<id>` groups. Pass `--summarize-existing` with the same filters and `--out-dir` to re-read existing sweep output without rerunning comparisons.

## Case-label classification convention

For trigger-profile and transform rows, classify the grouped summary before opening another one-off probe. Use `genValidProfileCaseLabel` or the fallback `transform:<id>` label as the grouping key, inspect one representative `binaryen.wat` / `starshine.wat` diff per label, and record the result in the row's `classifiedCaseLabels` array in [`parity-matrix.json`](./parity-matrix.json). Each entry records the label, transform id, representative failure directory, matrix status, agent classification, observed diff, next action, and evidence. The status is an agent judgment; the harness only reports mismatch mechanics.

The first classified count-3 smoke is `.tmp/oi-parity-sweep-summary-smoke` (2026-06-27):

- OI-D labels were mixed mismatches. The narrow implementable candidate was literal double-`eqz` final folding, where Starshine left `i32.ne(1, 0)` after the double-`eqz` rewrite while Binaryen printed `i32.const 1`. A follow-up red-first pass test now covers literal `i32.const 1; i32.eqz; i32.eqz`, and the fresh `.tmp/oi-parity-sweep-oi-d-double-eqz-fix-smoke` count-3 sweep confirms that drift is gone. The remaining OI-D raw mismatches are measured Starshine-win candidates in the sampled representatives: Starshine's pure constant compare/sign-extension folds reduce canonical wasm by 5 bytes in the commuted case and by 2 bytes in each direct/local-carried case, with both Binaryen and Starshine outputs validating.
- OI-E `local-carried-mask` and `direct-sign-extension` are Starshine-win candidates: Starshine removes redundant sign extensions from local facts while Binaryen keeps the explicit opcodes in the representative output. OI-E `local-tee-add-zero` is mixed because the commuted-operands transform also introduces OI-D scalar constant-compare drift.


## Dedicated GenValid smoke profiles

The matrix records the first pass-specific profiles and `src/validate/gen_valid.mbt` accepts them. Most remain smoke-level configs: each maps to a bounded Binaryen-oracle GenValid recipe with the right broad feature floor, but does not yet guarantee an OI-specific trigger in every generated module. `pass-oi-default-scalar`, `pass-oi-local-facts`, and `pass-oi-ref-gc` are now one step stronger: they emit deterministic seed-indexed trigger smoke modules for OI-D/OI-E/OI-I. Seeds are mapped modulo three to direct, local-carried/local.tee, mask/commuted-style, or reference branch-cast forms, so `--count 3 --seed 0x5eed` exercises three distinct trigger classes. The GenValid manifest now records each selected OI trigger as `profile_case_label`, and compare-pass copies that into `result.json` `genValidProfileCaseCounts`, `summary.json` strategy counters, and `cases.jsonl` `genValidProfileCaseLabel` fields for mechanical family triage. They still need randomized trigger-biased constructors and existing-producer wrappers before family signoff. `pass-oi-ref-gc` is currently a non-descriptor OI-I trigger smoke because the prior broad descriptor-bearing profile caused wasm-tools baseline validation failures in compare-pass; OI-J descriptor/exactness/TNH/IIT needs a separate compatible profile or oracle path.

| Profile | Family | Intended surface |
| --- | --- | --- |
| `pass-oi-default-scalar` | OI-D | scalar arithmetic, compares, `eqz`, shifts, float/conversion, and direct/local-carried/local.tee spellings |
| `pass-oi-local-facts` | OI-E | `LocalScanner`-style maxBits, signExtBits, unsigned max, known nonnegative/nullness, local.tee carriers, and single-use facts |
| `pass-oi-boolean-select` | OI-F | boolean contexts, `if`/`select`, ternary shells, identical arms, effect/trap/no-reorder boundaries |
| `pass-oi-memory-bulk` | OI-G | memory32/memory64 load/store offsets, stored values, tiny copy/fill, zero-size trap-mode boundaries |
| `pass-oi-call-ref` | OI-H | `call_ref` / `return_call_ref` target directization and argument localization boundaries |
| `pass-oi-ref-gc` | OI-I, blocked seed for OI-J+ | non-descriptor reference basics: direct null/test/cast/equality, local-carried null/test/cast, and branch-cast boundaries; descriptor/exactness/TNH/IIT, GC constructors/fields/arrays/defaults, and GC atomics still need split profiles |
| `pass-oi-tuple` | OI-M | tuple extraction, selected lanes, multivalue, tuple-scratch, and effectful/trapping sibling boundaries |

First generator-quality priority remains scaling the new seed-indexed `pass-oi-default-scalar` / `pass-oi-local-facts` / `pass-oi-ref-gc` trigger smokes into randomized family constructors, then replacing the remaining smoke configs for `pass-oi-memory-bulk` and `pass-oi-tuple` with trigger-biased constructors. OI-J should not reuse the current `pass-oi-ref-gc` smoke as descriptor evidence; it needs a descriptor/exactness-specific lane once the compare-pass oracle can accept those inputs.

`oi-local-tee-wrapped` is now implemented as a smoke transform in `src/fuzz/metamorphic.mbt`: it adds one i32 local to each defined function, prepends `i32.const` / `local.tee` / `drop` plus a `local.get` / `i32.add` / `drop` use, and leaves the original body after that stack-neutral prefix. `oi-commuted-operands` is now implemented as a smoke transform in `src/fuzz/metamorphic.mbt`: it prepends stack-neutral i32 constant pairs for `i32.add`, `i32.and`, and `i32.eq` in both operand orders before each defined function body.

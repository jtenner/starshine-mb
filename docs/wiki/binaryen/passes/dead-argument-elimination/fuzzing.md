# `dead-argument-elimination` Fuzzing Profile

This page tracks the pass-owned GenValid lanes for the plain `dead-argument-elimination` / `dae` audit and the related `dae-optimizing` closeout lane.

## Dedicated profiles

The DAE profile registry now has a replayable aggregate plus focused leaf names:

- Aggregate: `dead-argument-elimination`
- Aliases: `dae`, `dae-all`, `dae-closeout`, `dead-argument-elimination-all`, `dead-argument-elimination-closeout`, `pass-dae-closeout`
- Leaves:
  - `dae-unused-params`
  - `dae-effectful-args`
  - `dae-constant-args`
  - `dae-unused-returns`
  - `dae-arg-type-refinement`
  - `dae-return-type-refinement`
  - `dae-unseen-call-boundary`
  - `dae-tail-call-boundary`
  - `dae-localization`
  - `dae-convergence`
- Optimizing sibling: `dae-optimizing`
- Optimizing alias: `dae-optimizing-closeout`

The aggregate currently samples the ten direct-DAE leaves with weights biased toward the core positive rewrite families: unused params and constant args have weight `3`, effectful args and unused returns have weight `2`, and the refinement/boundary/localization/convergence leaves have weight `1` each. The optimizing sibling remains a separate profile so direct DAE signoff and nested-cleanup signoff can be reported independently.

## Generator envelope

`gen_valid_dead_argument_elimination_profile_config(profile)` keeps DAE cases small and direct-call oriented:

- no imports by default,
- 3 defined functions for most leaves,
- 4 defined functions for the effectful-argument and localization leaves,
- 5 defined functions for convergence and optimizing contexts,
- scalar-only value types by default,
- GC/subtyping enabled only for argument/return type-refinement leaves,
- tail calls enabled only for `dae-tail-call-boundary`,
- table/element/indirect-call surfaces enabled only for `dae-unseen-call-boundary`,
- globals enabled only for `dae-effectful-args` and `dae-localization`.

The first slice wired the profile names, aliases, composite membership, config helper, public API snapshot, and registry tests. Follow-up slices filled the body-template layer:

- `dae-unused-params` bypasses the generic body generator and emits a deterministic three-function module where private helper function `1` has three `i32` parameters, uses only parameter `0`, and is called directly from the exported entry and an extra private caller. This gives the direct DAE lane a stable unused-parameter rewrite trigger.
- `dae-effectful-args` bypasses the generic body generator and emits a deterministic four-function module with one mutable private `i32` global. Private helper function `1` has signature `(i32, i32) -> i32`, uses only parameter `1`, and both direct callers supply parameter `0` from private producer function `2`, whose body performs `global.set` before returning an `i32`. This gives the direct DAE lane a stable effectful-unused-actual preservation trigger.
- `dae-constant-args` now also bypasses the generic body generator and emits a deterministic three-function module where private helper function `1` has signature `(i32, i32) -> i32`, uses both parameters, and both direct callers pass the same second `i32.const 77` actual while varying the first actual. This gives the direct DAE lane a stable same-constant-argument materialization/removal trigger.
- `dae-unused-returns` now bypasses the generic body generator and emits a deterministic three-function module where private helper function `1` has signature `(i32, i32) -> i32`, both direct callers immediately `drop` the helper result before returning their own constants, and the helper computes one `i32` result from its parameters. This is the generated single-result positive DAE-F leaf: Binaryen and Starshine both remove the dropped private helper result in the refreshed focused lane. The prior two-result template is superseded as a profile trigger but remains recorded below as a sampled multi-result tuple/local-spill avoidance residual from `.tmp/pass-fuzz-dae-direct-smoke-100-after-exact`.
- `dae-arg-type-refinement` now bypasses the generic body generator and emits a deterministic three-function GC module with a non-final empty base struct type, a final leaf subtype, and a private helper whose parameter is typed as the broad nullable base while both owned direct callers pass `struct.new_default` values of the leaf subtype. The helper consumes the parameter with `ref.is_null`, so this leaf targets argument type refinement rather than unused-parameter deletion. A 2026-07-05 generator fix made the base non-final after `wasm-tools` rejected the subtype leaf when the base used the shorthand final `comp_type` form.
- `dae-return-type-refinement` now bypasses the generic body generator and emits a deterministic three-function GC module with the same non-final base/final leaf subtype pair. A private helper is declared as returning the broad nullable base type but always produces `struct.new_default` of the leaf subtype, and both owned direct callers immediately consume the helper result with `ref.is_null`. This gives the direct DAE lane a stable return type-refinement trigger.
- `dae-tail-call-boundary` now bypasses the generic body generator and emits a deterministic three-function module where private helper function `1` has signature `(i32, i32) -> i32`, uses only parameter `1`, and is reached from the exported entry plus an extra private caller only through `return_call`. This originally targeted a presumed tail-call bailout; the 2026-07-05 direct compare instead showed Binaryen rewrites this return-call-only shape. A later 2026-07-05 red-first pass fixture fixed Starshine's direct `return_call` call-fact collection and callsite operand repair; remaining sampled tail-leaf compare drift is only Binaryen retaining one unused body local after the signature prune while Starshine omits it.
- `dae-unseen-call-boundary` now bypasses the generic body generator and emits a deterministic three-function module with one `funcref` table and one active element segment containing `ref.func 1`. The exported entry directly calls private helper function `1`, private function `2` invokes the same helper signature through `call_indirect`, and the helper uses only parameter `1`. This gives the direct DAE lane a stable unseen-call/ref.func/table/indirect-call bailout trigger that otherwise looks like an unused-parameter rewrite candidate.
- `dae-convergence` now bypasses the generic body generator and emits a deterministic five-function module where private wrapper function `1` forwards both parameters to private leaf function `2`, while the leaf uses only parameter `0`. The wrapper's second parameter is initially syntactically used only as the leaf's dead actual, so pruning the leaf signature unlocks a later wrapper-signature prune. This gives the direct DAE lane a stable convergence/stale-analysis retry trigger without tail-call or unseen-call boundaries.
- `dae-localization` now bypasses the generic body generator and emits a deterministic four-function module with one mutable private `i32` global. Private helper function `1` has signature `(i32, i32) -> i32`, uses only parameter `1`, and both direct callers supply parameter `0` from a value-producing block that calls private side-effect producer function `2`, whose body performs `global.set` before returning an `i32`. This gives the direct DAE lane a stable hard effectful operand that must be localized or otherwise preserved before the unused-parameter rewrite can safely complete.
- `dae-optimizing` now also bypasses the generic body generator and emits a deterministic five-function module where private helper function `1` has signature `(i32, i32) -> i32`, uses only parameter `1`, and leaves an identity `local.get 1; i32.const 0; i32.add` tail in the touched helper body while four direct callers make parameter `0` removable. This gives the optimizing sibling a stable nested-cleanup trigger after the core DAE signature rewrite changes the helper.

# Generator Coverage Ledger

Starshine's fuzzer generator widening work uses a durable coverage ledger so generator smoke and CI runs can report both covered and intentionally missing surfaces without treating every future family as a hard gate.

## Current contract

- `src/validate/gen_valid.mbt` owns the explicit named generator profile taxonomy. `gen_valid_profile_by_name(...)`, `GenValidConfig::for_profile(...)`, and `GenValidConfig::profile(...)` expose stable profile names for natural-small, natural-wide, natural-deep, coverage-forced-portable, coverage-forced-all-features, binaryen-oracle-portable, binaryen-oracle-relaxed-simd, mutation-seed-small, mutation-seed-rich, pass-fuzz-stress, validator-stress, pass-cleanup, pass-dae, pass-inlining, pass-memory, pass-gc-ref, pass-control, no-imports, imports-heavy, GC-heavy, SIMD-heavy, relaxed-SIMD, memory-heavy, control-heavy, topology-heavy, and pathological-valid. Legacy labels such as `natural`, `coverage-forced`, `portable-coverage-forced`, and `binaryen-oracle-coverage-forced` remain accepted aliases for compatibility. The first `[FUZ]1002` typed body-generator slice is gated by `allow_typed_body_generator` and currently enabled by the `control-heavy` profile, where it emits a stack-planned `local.tee`, typed value block, typed value `if`, and branch-bearing void block before the broader natural tail. The `[FUZ]1003` topology-heavy slice is a coverage-forced profile with larger type/import/function/table/memory/global/tag/segment budgets, forced section presence, at least ten total function indices on generated modules, multiple import module names, repeated valid exports of the same item under distinct names, imported-function re-exports, nonzero table/memory/global/tag resource-index exports when those spaces are available, nonzero active element/data segment targets, multiple binary-supported leading non-name custom sections plus the structured name section, and explicit variants that avoid the conventional `main` export name or omit exports entirely. The first `[FUZ]1035` budget slices add optional post-generation instruction-count, expression-depth, and estimated module byte-size budgets to `GenValidConfig`; candidates that exceed a budget are retried and final failures carry `GenValidFailure.budget_stop` with a clear budget-exceeded diagnostic. `GenValidConfig` also now bounds generated name length and data/custom segment payload size so stress profiles can approach large-shape limits without silently producing unbounded metadata. The `pathological-valid` profile raises topology/body limits for stress generation while retaining bounded instruction-count, expression-depth, byte-size, name-length, and segment-payload caps.
- `src/validate/validate.mbt` owns the public ledger API: `validate_valid_feature_ledger(stats, floors)` returns one row per intended surface with a stable key, label, observed count, required minimum, and status.
- `check_validate_valid_feature_floors(stats, floors)` still fails only for floors explicitly listed by a profile or caller. Missing future FZG families are reported as `MissingOptional` until a profile adds a nonzero floor. String-label helpers `validate_valid_feature_floor_by_label(...)` and `validate_valid_feature_actual_count_by_label(...)` are available for harnesses that should key user/profile filters by stable labels instead of enum constructors.
- `src/fuzz/main.mbt` exposes `emit_gen_valid_batch_artifacts_until_features(...)` for local coverage-guided GenValid selection. It walks a deterministic bounded candidate stream, skips candidates that hit excluded feature keys, keeps only modules that advance a missing required floor (or the first `count` modules when no floors are requested), and reports selected artifacts, manifest rows, aggregate stats, remaining floor failures, total attempts, and skipped candidates. This is the initial transform registry surface; pass-fuzz integration remains future `[FUZ]1036` work. `[FUZ]1034` has started the deterministic PRNG stream-split audit: `src/validate/gen_valid.mbt` now defines stable stream labels for module topology, types, funcs, bodies, memories, tables, data, elems, globals, tags, start-section choice, invalid strategy choice, invalid mutation choice, text mutation choice, and harness shuffling, plus a fixed root-seed-to-stream-seed derivation helper. GenValid batch JSON manifests now include `stream_derivations` so repro manifests can cite which named stream owns a random choice. The first migrated generation path is the explicit root-seed batch API: `gen_valid_module_result_from_seed(...)` derives named streams per candidate, routes type/result and GC type-topology choices through the `types` stream, function-count, function-signature assignment, and local-declaration shape choices through the `funcs` stream, body statement/control/callable-helper choices through the `bodies` stream, memory/table/data/element shape choices and element `ref.func` helper choices through their matching streams, global/tag/start choices through their dedicated streams, and import/export/name-section topology choices explicitly through the module-topology stream, while keeping the legacy `RandomState` API available for callers that need the old monolithic seed behavior. The invalid-AST fuzzer now derives each strategy/seed-profile base generation seed from the `invalid-mutation-choice` stream plus stable strategy/profile salts and feeds that seed through `gen_valid_module_result_from_seed(...)`, so adding or rescheduling unrelated invalid strategies no longer advances a shared monolithic GenValid stream for existing strategy/profile seeds. The binary-invalid fuzzer now uses the same `invalid-mutation-choice` stream plus stable strategy/profile salts for each attempt instead of advancing one monolithic binary invalid `RandomState`; AST and binary invalid repro metadata also persists the exact `attempt_seed` used for the per-strategy/profile GenValid root seed alongside the owning stream label/seed. The dynamic invalid-text GenValid-WAT lane similarly derives each strategy/attempt base-module seed from the `text-mutation-choice` stream plus stable strategy salts before calling `gen_valid_module_result_from_seed(...)`, so adding a new text mutation no longer perturbs existing dynamic text strategy specimens; its suite details now expose the `text-mutation-choice` stream label and derived stream seed for repro audits. GenValid batch candidate root seeds now derive from the `harness-shuffling` stream before per-candidate salting, so changing batch candidate scheduling is separated from the root seed and from the generator's internal module-topology/type/function/body streams. `parse_gen_valid_batch_manifest_stream_derivations(...)` roundtrips the manifest `stream_derivations` array back into `{ label, derived_seed }` records so tests and future repro tooling can detect stream-map schema drift instead of only asserting raw JSON substrings.

## How to use the ledger

Use this page as the bridge between generator work, fuzz-runner commands, and validation gates:

1. **When adding a valid-generator surface**, add a stable `ValidateValidFeatureKey`, record it in `validate_valid_feature_ledger_keys()`, update `GenValidFeatureStats` / feature scanning, add a coverage-forced generator shape, and land a focused validation-anchor test in [`src/validate/validate.mbt`](../../../src/validate/validate.mbt). Do not raise smoke/CI/stress floors until the coverage-forced shape validates deterministically.
2. **When adding an invalid lane**, keep the deterministic mutation/repro path in the invalid AST or binary helpers, make the expected validation family explicit, and link the replay contract back to [`../validate/fuzz-hardening.md`](../validate/fuzz-hardening.md) plus the focused family/stage/artifact rules in [`../validate/diagnostics-and-invalid-repro.md`](../validate/diagnostics-and-invalid-repro.md) rather than treating it as a valid-generator floor.
3. **When adding WAST text generation**, update [`wast-arbitrary-parity-plan.md`](wast-arbitrary-parity-plan.md) and keep `src/wast/arbitrary.mbt` independent from `gen_valid`; WAST mirrors the FZG vocabulary but is a parser/printer surface, not a typed module-validity oracle. For scalar numeric, ordinary control, reference, GC aggregate/i31, string helper, exception, and SIMD shapes, also keep [`../wast/numeric-instruction-authoring.md`](../wast/numeric-instruction-authoring.md), [`../wast/control-flow-authoring.md`](../wast/control-flow-authoring.md), [`../wast/reference-instruction-authoring.md`](../wast/reference-instruction-authoring.md), [`../wast/gc-aggregate-instruction-authoring.md`](../wast/gc-aggregate-instruction-authoring.md), [`../wast/string-instruction-authoring.md`](../wast/string-instruction-authoring.md), [`../wast/exception-tag-authoring.md`](../wast/exception-tag-authoring.md), and [`../wast/simd-authoring.md`](../wast/simd-authoring.md) aligned.
4. **When using `bun fuzz compare-pass`**, treat [`../tooling/pass-fuzz-compare.md`](../tooling/pass-fuzz-compare.md) as the workflow contract and `[FZG]029` failure metadata as the durable replay bridge: persisted failures should carry `failure-metadata.json` plus the copied input artifacts, while the generated-input batch still comes from `src/fuzz --emit-gen-valid-batch` through the wrapper described in [`../tooling/fuzz-runner.md`](../tooling/fuzz-runner.md).

The official WebAssembly module-validation rules are still the external semantic anchor for segment, table, memory, ref, and code-body validity; the local ledger's job is narrower: prove Starshine's generator can deliberately produce and measure those families, and prove each required profile fails only for the rows it explicitly requires.
